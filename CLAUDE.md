# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

Purple Shop — a client-side order/inventory management web app (Spanish UI). No build step, no bundler, no npm dependencies: plain HTML/CSS/JS served as static files, with Microsoft OneDrive/Excel integration via MSAL.js and Microsoft Graph.

## Commands

There is no build/compile/lint/test tooling configured (no npm dependencies in `package.json`). Development is "edit the file, reload the browser."

- Run locally: `npm start` (Python http.server on :8000) or `npm run serve` (npx http-server on :8000). Serving over HTTP (not `file://`) matters because the OneDrive redirect URI and `fetch("html/inventory.html")` in `index.html` require it.
- No test runner is wired up. `tests/unit-tests.js` is a hand-rolled assertion harness (see its own header comment) meant to be loaded in a browser page after all `src/` modules — there is currently no HTML file that does this for it. The existing `tests/*.html` files (`inventory-test.html`, `inventory-buttons-test.html`, `inventory-buttons-full-test.html`) are manual/visual test pages you open directly in a browser, not an automated suite.
- `tests/security/SECURITY_TESTS.js` is meant to be pasted into the browser devtools console against a loaded page (per `docs/STRUCTURE.md`), not run headlessly.

## Architecture

**No modules/bundler — everything is global.** Each file in `src/` defines things and attaches them to `window` (e.g. `window.Config`, `window.AuthManager`, `window.InventoryManager`, `window.EventBus`). There is no `import`/`export` or build step, so **script load order in `index.html` is load-bearing** — a script that references `window.SomeManager` must be loaded after the file that defines it. `index.html` documents the required order explicitly in comments:

1. External libs (MSAL browser SDK via CDN)
2. Infrastructure with no dependencies: `src/utils/event-bus.js`, `notification-service.js`, `logger.js`, `performance.js`, `sanitize.js`, `validation.js`
3. `src/config/config.js` (depends on utils)
4. App modules depending on config+utils: `src/modules/auth.js`, `inventory.js`, `excel.js`, `ui.js`, `forms.js`, `order-form.js`
5. Product-inventory tab module: `src/core/inventory.js`, `src/core/inventory-ui.js`
6. `src/utils/form-optimizations.js`
7. `src/main.js` — the orchestrator, must load last

When adding a new module, wire it into this same list at the right position rather than introducing a separate loading mechanism.

**`src/main.js` (`window.App`)** is the entry point: on `DOMContentLoaded` it validates required modules are present (`Config`, `UIManager`, `InventoryManager`, `FormManager` are required; `AuthManager`, `ExcelManager`, `SecurityUtils`, `ValidationUtils` are optional and only warn if missing), initializes them in sequence, wires up top-level event listeners (login/logout, search, export/import, sync), sets up tab navigation, and starts a 5-minute auto-sync interval with Excel when authenticated (cancelled on logout/cleanup).

**Inter-module communication** goes through `window.EventBus` (simple pub/sub — `on`/`once`/`off`/`emit`) rather than modules calling each other directly, to reduce coupling (see `src/utils/event-bus.js`). Prefer emitting/listening on the bus over reaching into another module's globals for new cross-module behavior.

**`src/core/legacy/app.js` is superseded** — it was the original monolithic app.js before the "Fase 2: Modularización" refactor split things into `src/modules/` + `src/main.js`. It is not loaded by `index.html` and should be treated as historical reference only, not a place to make changes.

**Config (`src/config/config.js`)**: MSAL/Azure AD config, Excel column mapping, UI/validation/logging constants, all under `window.Config`. Client ID is read at runtime via `getEnvVar()`, which checks `window.ENV[key]` first, then `localStorage.getItem('env_' + key)`. `.env.example` documents the expected `VITE_*` variable names, but nothing loads an actual `.env` file (no Vite is wired up despite the variable naming). The real injection point is `src/config/config.local.js`, loaded by `index.html` right before `src/config/config.js`. **This file is intentionally committed to the repo, not gitignored** — a public-client SPA's Azure AD Client ID isn't a secret (there's no client secret in this flow; MSAL Browser uses PKCE), it's meant to ship in browser-visible JS, and the static host (GitHub Pages) has no build step to inject it otherwise. `src/config/config.local.example.js` is the template for setting one up from scratch. If `config.local.js` is ever deleted, the browser logs one harmless 404 for that `<script src>` and the rest of the app still loads — login/Excel features just stay disabled.

**Security utilities (`src/utils/sanitize.js`, `validation.js`)**: `sanitizeText()` and `sanitizeHTML()` (DOMParser-based allowlist rebuild, drops all attributes) exist specifically to avoid `innerHTML`-based XSS; `ValidationUtils.validateField()` / `validateOrderData()` centralize field/order validation. Any new code that renders user-controlled strings into the DOM or accepts form input should go through these rather than ad hoc string interpolation into `innerHTML`.

**Pages**: `index.html` is the single-page app shell (tabs: Nuevo Pedido, Clientes, Pedidos, Reportes, Inventarios — all client-side tab switching, no routing). The Inventarios tab's markup lives in `html/inventory.html` and is fetched and injected into `#inventoryModule` at runtime, then `initInventoryUI()` (from `src/core/inventory-ui.js`) is called to wire it up — so that module's DOM doesn't exist until after this fetch resolves. `html/INTEGRACION_FASE2.html` is an orphan file — nothing in the app (`index.html`, any script) loads or links to it; it's only mentioned by stale files under `docs/`.

**Excel/Graph integration is hand-rolled `fetch`, not the Graph SDK.** `src/modules/excel.js`'s `ExcelManager` is named around a `graphClient` field, but that field is explicitly always `null` — every call to Microsoft Graph is a raw `fetch` to `https://graph.microsoft.com/...` with `Authorization: Bearer <token>` from `AuthManager.getToken()`. Follow that same raw-fetch pattern for new Graph calls rather than introducing the `@microsoft/microsoft-graph-client` SDK. Reads/writes target the worksheet by name (`workbook/worksheets('{sheetName}')`, URL-encoded), where `{sheetName}` comes from `Config.excelConfig.sheetName` (`'Pedidos'` by default) — this must match the actual tab name in the user's real `.xlsx` file in OneDrive, not necessarily "Sheet1". `writeToExcel()` computes its range address (`A1:{col}{row}`) to exactly match the dimensions of the `values` matrix being sent, because Graph's range-update API requires the address and payload dimensions to agree.

**Multi-user shared workbook, not per-user files.** All Graph item endpoints use `/drives/{driveId}/items/{itemId}/...`, never `/me/drive/...` — this is what lets a second, independently-authenticated account read/write the *same* Excel file that lives in someone else's OneDrive (as long as that owner has shared it with them). `ExcelManager.ensureExcelFile()` branches on `Config.excelConfig.sharedDriveId`/`sharedItemId`: if both are set (in `config.local.js`), it resolves that exact item via `_getItem()` and fails loudly if the current account has no access to it; only when they're unset does it fall back to the original find-or-create-by-name behavior scoped to the caller's own drive root. When onboarding a new collaborator, the file owner must explicitly share the workbook with that person's Microsoft account (or the sync will 403/404) — sharing is a OneDrive-side action, this app has no UI for it.

**⚠️ `writeToExcel()`/`syncInventory()` must NOT be pointed at the real business workbook as-is — it will destroy data.** The owner's actual OneDrive Excel file (filename "REFERENCIAS" — ask the repo owner for its `driveId`/`itemId` if you need to work on this; deliberately not written down here since this file is public) is *not* a flat one-row-per-order table. It's hand-maintained by the business owner's manager: one worksheet per month (tab names like `SEPT 2025`, `OCT 2025`, ... `JULIO 2026` — inconsistently abbreviated, so don't try to compute the current month's tab name from a date; if this is revisited, detect it as the last/rightmost worksheet instead), and *within* each monthly sheet, variable-height blocks — one per client — each with its own header row (`REF`/`CLIENTE`/`DESCRIPCION`/`CANTIDAD`/`PRECIO TOTAL`), one row per line item, then `SUBTOTAL`/optional `ENVIO`/`TOTAL` rows, then a blank row before the next client's block. `writeToExcel()` as it exists blindly overwrites `A1:{col}{row}` with `InventoryManager`'s own flat 14-column schema — running it against this file would clobber the real data on the very first sync. `Config.excelConfig.sharedDriveId`/`sharedItemId` are therefore left **blank** in `config.local.js` on purpose (2026-07-15). Don't repopulate those two config values, and don't call `syncInventory()`/`writeToExcel()` against the real business file, without first designing (and testing against a *copy* of the file) an importer/appender that understands this per-client-block structure. Until that exists, the app's order data stays local per-account (each signed-in user gets their own auto-created/found file, per the fallback path) — it does not reach the shared business workbook.

**Login allowlist is app-level, not an Azure feature — and its config is deliberately NOT committed.** Azure AD has no built-in way to restrict *personal* Microsoft accounts to a specific list (the "assignment required" feature only applies to accounts inside an admin-managed tenant, not consumer MSAs) — so `AuthManager` implements it itself. `Config.authConfig.allowedAccountHashes` holds SHA-256 hex hashes (never plaintext emails) of `email.trim().toLowerCase()` for each allowed account, read via `getEnvVar('VITE_ALLOWED_ACCOUNT_HASHES', '')` (comma-separated). Unlike the Client ID, this value is **not** set in the committed `config.local.js` — a hash of a real person's email is still personal data worth keeping out of a public repo (SHA-256 with no salt is crackable by dictionary attack against a suspected/known email, even though it doesn't reveal an unknown one from scratch). Instead it's meant to be set per-browser via `localStorage.setItem('env_VITE_ALLOWED_ACCOUNT_HASHES', '<hash1,hash2,...>')` — `getEnvVar()` already checks that key as a fallback. To compute a hash: `node -e "console.log(require('crypto').createHash('sha256').update('correo@ejemplo.com').digest('hex'))"`. Practical effect: on a browser/device where that `localStorage` key was never set, the allowlist is empty and login isn't restricted — acceptable given the next paragraph. `AuthManager._isAccountAllowed()` hashes the signed-in account's `username` (via `crypto.subtle`, only available in a secure context: `https://` or `localhost`) and checks membership; it's called both in `initialize()` (before restoring a cached MSAL session) and in `handleLogin()` (right after `loginPopup()` resolves, before `setActiveAccount()`). **This is a soft, client-side gate only** — anyone can read the app's JS and see the check. It exists to give unauthorized visitors a clean "no access" message, not to withstand a determined attacker. The actual enforcement boundary is Microsoft's own permission check on the shared Excel file (`/drives/{id}/items/{id}` 403s for anyone the owner hasn't shared it with) — that's server-side and can't be bypassed by editing client JS. Don't add new sensitive checks to this codebase assuming client-side logic is a real security boundary; it isn't.

### Fixed gotcha (for history — don't reintroduce): "Nuevo Pedido" form had no real save path

Until 2026-07-15, the multi-product order form's interactivity (category-specific dynamic fields, per-line/grand total calculation, product quick-codes, client autocomplete, add/remove product row) only existed in `src/core/legacy/app.js` — the file `index.html` explicitly does not load. None of the currently-loaded modules attached a `submit` listener to `#orderForm`, so clicking "Guardar Pedido" just did a native no-op form submit (page reload); the inline `onchange`/`oninput` attributes in `index.html` (`updateProductFields(0)`, `handleProductCodeInput(0)`, `updateGrandTotal()`, `selectExistingClient()`, `checkClientExists()`, `switchTab(...)`) referenced globals that didn't exist outside legacy code, so most of the tab was non-functional (console errors on category change, totals never recalculated, "+ Agregar Producto" did nothing).

Fixed by porting that logic into `src/modules/order-form.js` (`window.OrderFormManager`), adapted to the modular data model rather than copied verbatim:
- `InventoryManager`'s `addOrder()` model is **one entry per product line** (flat `clientName`/`phoneNumber`/`email`/`address`/`productName`/`quantity`/`price`/`totalPrice`/`discount`/`shippingCost`/`notes`/`orderDate`), not one entry per multi-product order. `OrderFormManager._handleSubmit()` collects the whole form once, then calls `InventoryManager.addOrder()` once per product row, tagging every line with a shared `orderGroupId` (`crypto.randomUUID()`) so a multi-product order can still be reassembled later by whoever reads `InventoryManager.getAll()`.
- HTML field names don't match the canonical order-data keys 1:1 (`clientPhone` in the form → `phoneNumber` on the order object, `clientAddress` → `address`) — get this mapping wrong and validation/Excel sync silently misfire.
- The client-search datalist is populated from `InventoryManager.getAll()`'s existing distinct clients (simple, local, no extra Excel file) rather than legacy's separate `Clientes.xlsx` read/write — that's out of scope for this fix; the "Clientes" tab's own cloud sync (if any) is unrelated code.
- `window.updateProductFields`, `handleProductCodeInput`, `updateGrandTotal`, `selectExistingClient`, `checkClientExists`, `switchTab` are re-exposed as thin global wrappers delegating to `OrderFormManager`, purely so the *existing* inline attributes on product row 0 in `index.html` keep working without editing that markup; rows added via "+ Agregar Producto" are built and wired with `addEventListener` instead (no inline attributes) in `OrderFormManager._buildProductRow()`.
- Also fixed in the same pass: `InventoryManager.addOrder()` used to unconditionally overwrite `orderDate` with `new Date().toISOString()` even when the form's date picker had a real value — it now only defaults to "now" when no `orderDate` was supplied.

If you touch the order form again, test by actually submitting a multi-product order in the browser and checking `window.InventoryManager.getAll()` in devtools — this area has no automated test coverage (see Commands section) and broke silently once already. Verified working end-to-end via real browser automation on 2026-07-15 (dynamic category fields, conditional fields, totals with discount/shipping, quick-code save/autofill, add/remove rows, multi-line submit grouped by `orderGroupId`, client autocomplete, graceful handling of a failed Excel sync).

While browser-testing the above, two more pre-existing, unrelated bugs surfaced and were fixed in the same pass:
- **`src/core/inventory-ui.js` had three separate instances of a "missing opening line" bug** — a `.map(x => \`...\`).join('')` (or equivalent) whose opening line (`container.innerHTML = items.map(item => \``, `const html = \`...`) had been deleted, leaving orphaned template-literal bodies. This was a hard **parse-time** `SyntaxError`, so it silently broke the entire file — nothing in it executed, including `initInventoryUI`, which is why the Inventarios tab never rendered its injected HTML (`html/inventory.html` loaded, but `initInventoryUI` was undefined). Fixed in `renderMovementsTable()`, `loadCategories()`, and `showInventoryReport()`. Also removed a call to `initFormHandlers()` in `initInventoryUI()` — that function is never defined anywhere in the codebase (dead call from an incomplete refactor); the form-submit wiring it would have done is already covered by `attachInventoryButtonListeners()` in the same file.
- **`index.html` had ~75 instances of literal, undecoded JS-style unicode escapes** (`ó`, `á`, `⚡` — i.e. the six-character string `\`, `u`, `0`, `0`, `f`, `3`, etc., not an actual "ó") sprinkled through visible UI text and HTML comments, rendering as garbage like `Sección de códigos guardados` in the browser instead of "Sección de códigos guardados". Likely from a previous edit that wrote JSON/JS-escaped text directly into HTML instead of decoding it first. Fixed with a straight literal-string replacement across the whole file (`á`→á, `ó`→ó, `⚡`→⚡) — if you see similar garbled text elsewhere in this codebase, check for the same class of bug (`grep -o '\\\\u[0-9a-fA-F]\{4\}' <file>`) before assuming it's a rendering/font issue.
- Note for testing this app locally: Chrome aggressively disk-caches `<script src>` files served by `python -m http.server` (which sends no cache-control headers), so after editing a `src/` file, `Ctrl+Shift+R` was not always sufficient to see the change reflected — opening a fresh tab (or a cache-busting `?query` on the navigated URL) was what reliably picked up the new file content during this session's testing.

### Fixed gotcha (for history — don't reintroduce): two classes both named `InventoryManager`

`src/modules/inventory.js` (orders/pedidos) and `src/core/inventory.js` (product stock for the Inventarios tab) used to each declare their own top-level `class InventoryManager { ... }`. Classic (non-module) `<script src>` tags in the same document share one global lexical scope for `let`/`const`/`class` (unlike `var`/`function`, which attach to `window`), so the second declaration silently redeclared the same binding and would throw `SyntaxError: Identifier 'InventoryManager' has already been declared` when `src/core/inventory.js` loaded — breaking that entire script (and the product-inventory tab) since it loads after `src/modules/inventory.js`. Fixed by renaming the class in `src/core/inventory.js` to `ProductInventoryManager` (its instance is still exposed only as the script-local `const inventory`, which is what `src/core/inventory-ui.js` actually reads — nothing external depended on the class name itself). **Don't reuse a class name across two `<script>`-loaded files anywhere in this codebase — there's no module scoping to protect you.**

**Docs housekeeping**: `docs/` contains a large number of historical status/summary/phase-completion documents (Spanish and English) from past refactor phases; several (e.g. `docs/STRUCTURE.md`, `docs/ESTRUCTURA_ORGANIZADA.md`) describe an older `js/` folder layout that no longer exists — the current source layout is under `src/` as described above. `README.md`'s own "Estructura del Proyecto" tree and its Excel filename (`Pedidos.xlsx`) are similarly stale — the actual configured filename is `PedidosInventario.xlsx` (`src/config/config.js`'s `excelConfig.fileName`). Don't trust folder-structure or config claims in docs without checking the actual source.
